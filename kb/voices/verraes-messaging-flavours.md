# Verraes — Messaging Flavours

Verraes has a more nuanced taxonomy of messages than the basic
command/query/event trio. In ideation conversations, naming the exact flavour
of a message often reveals where a boundary should or shouldn't be.

## The flavours (beyond Command/Query/Event)

### Commands
A message that asks a specific receiver to do a specific thing.

- **Imperative Command** — "Place the order." Single target. May succeed or
  fail. Verb in imperative.
- **Policy Command** — "When X, do Y." Conditional, often generated by a
  policy reacting to an event. Same shape as Imperative but triggered by the
  system rather than a user.

### Queries
A message asking for information.

- **Query** — "What is the status of order X?"

### Events
A message stating that something happened. Past tense. Multiple receivers
possible.

- **Domain Event** — A business-meaningful fact. "OrderPlaced". Named in
  business language. Emitted by an aggregate (usually).
- **Integration Event** — A fact crossing a BC boundary. Usually a
  downstream-friendly projection of a domain event. "OrderPlacedV2" with a
  flatter/stable schema.
- **Notification Event** — Carries no state; just announces. Receivers
  query back for details.
- **Event-Carried State** — Carries enough state to be consumed without
  callback.

(The state/notification axis maps to
`kb/canonical/fowler-event-driven-flavours.md`.)

### Other message types Verraes surfaces

- **Request / Response** — a pair. The request is Query-shaped; the response
  is typed data.
- **Reminder / Scheduled** — self-addressed future command ("at 18:00, close
  the auction").
- **Confirmation / Rejection** — responses to a command; often carry domain
  semantics ("OrderRejected — reason: fraud").
- **External Event** — from outside any BC (a webhook, a sensor reading).
  Needs an ACL to translate into a Domain Event the system uses.

## Why this matters in ideation

- In Event Storming, teams often call a sticky "Event" when it's actually a
  **Command** issued reactively. Naming the flavour exposes the confusion.
- Integration Events with the same name as Domain Events is a **coupling
  smell**: downstream BCs pinned to upstream implementation. Separate them
  explicitly (published language).
- A "Notification Event" that nobody actually queries back from is not
  producing value — probably should be state-carrying or removed.

## Heuristics Verraes offers

1. **Past tense for events, imperative for commands** — non-negotiable. If
   you can't name it in past tense, it's not an event.
2. **Domain Events inside the BC; Integration Events at the boundary.** Never
   let a Domain Event's internal structure become the published schema —
   changes become breaking.
3. **Commands across BC boundaries are suspicious.** Usually it's a sign of
   over-integration — the downstream BC is doing a job that belongs upstream.
   Prefer events as the cross-BC medium.
4. **Scheduled/Reminder messages deserve their own modelling.** Time as an
   actor. See `kb/voices/verraes-temporal-modelling.md`.

## How this file should be cited

When classifying messages on an event storm or in a BC canvas
(`kb/ceremonies/bounded-context-canvas.md` § 5–6):

> Per `kb/voices/verraes-messaging-flavours.md`, this looks like an
> Integration Event (crossing BC boundary). It should have its own stable
> schema, distinct from the Domain Event it was derived from.

## Interrogator pushbacks powered by this

- Sticky labelled "Event" but content is imperative → reclassify as Command.
- Command crossing BC boundary → **push back**: why not an event? Is the
  upstream doing someone else's job?
- Integration Event with no separate schema from Domain Event → warn about
  coupling.
- A "Notification Event" nobody acts on → remove or convert.
